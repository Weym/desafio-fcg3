- Impact: Zero backend functionality exists. Nothing in the stack (FastAPI, LangChain, MCP Server, PostgreSQL) is implemented. All planned API endpoints, AI agent, RAG pipeline, and WhatsApp webhook are documentation-only.
- Fix approach: Begin implementation following the Vertical Slice Architecture described in `README.md`, starting with the FastAPI scaffolding, database models, and auth feature slice.

- Fix approach: Update `README.md` to reflect the Python/FastAPI decision documented in `docs/architecture.md`.

## Security Considerations

### No `.env` file or environment variable configuration exists

- Risk: No `.env.example`, no environment variable documentation, and no mention of a `.env` file anywhere in the repository. The MCP service token (`MCP_SERVICE_TOKEN`) and all other secrets (WhatsApp token, database credentials, FCM service account, OpenAI API key) have no defined injection mechanism.
- Files: `docs/mcp.md` (references `MCP_SERVICE_TOKEN`), `docs/architecture.md` (references multiple services requiring auth)
- Current mitigation: Nothing in place. Backend is not implemented, so no secrets are exposed yet.
- Recommendations: Create `.env.example` with placeholder values for all required secrets. Add `.env` to `.gitignore` (currently only `node_modules/` is in `backend/.gitignore`). Document required env vars in `README.md`.

### WhatsApp webhook signature validation is documented but not implemented

- Risk: `docs/architecture.md` documents that the webhook must validate `X-Hub-Signature-256` to authenticate incoming WhatsApp requests. Without this, any caller can trigger the webhook and cause the AI agent to process arbitrary messages.
- Files: `docs/architecture.md`, `backend/src/features/` (empty)
- Current mitigation: Not applicable (backend not implemented). The risk materializes when the endpoint is built without the validation.
- Recommendations: Implement HMAC-SHA256 validation against `WHATSAPP_APP_SECRET` as the first step in the webhook handler, before any database writes.

### MCP Service Token is a static shared secret

- Risk: The `X-Service-Token` used for MCP-to-API calls is a single static token that never rotates. Any process that obtains this token has full access to all MCP-marked endpoints (grades, enrollments, documents, appointments) without student-level authorization.
- Files: `docs/mcp.md`
- Current mitigation: Token is documented to live only in environment variables, never in source code.
- Recommendations: This is an acceptable MVP risk. Post-MVP should consider signed short-lived tokens or mutual TLS between internal services.

### `document.file_url` is a plain VARCHAR stored directly in the database

- Risk: Document files are served via direct URLs in `documents.file_url` (documented as `https://storage.example.com/docs/...`). There is no mention of URL signing, access control, or expiry. Any person with a document URL can access the PDF without authentication.
- Files: `docs/database.md`, `docs/api.md` (`GET /documents/{id}`)
- Current mitigation: None documented.
- Recommendations: Use pre-signed URLs with expiration (e.g., S3 signed URLs, GCS signed URLs) when returning document links. Never store permanent public URLs.

### Verification code is a 6-digit numeric code — brute-forceable without rate limiting at transport layer

- Risk: The 3-attempt limit is enforced at the application level via the `attempts` column in `verification_codes`. If the rate limiting middleware is not implemented, an attacker can open parallel connections and exhaust codes across multiple requests concurrently before the single-attempt counter increments.
- Files: `docs/database.md` (`verification_codes.attempts`), `docs/api.md` (`POST /auth/verify-code`)
- Current mitigation: 3-attempt counter + 5-minute expiry documented. Actual enforcement depends on implementation.
- Recommendations: Implement per-IP and per-email rate limiting at the FastAPI middleware level in addition to the attempt counter, to prevent concurrent bypass.

## Performance Bottlenecks

### asyncio.create_task for background processing — no task failure visibility

- Problem: The webhook returns `200 OK` immediately and dispatches processing via `asyncio.create_task`. If the background task fails (LLM error, database error, WhatsApp API error), the failure is invisible to the student — they simply never receive a reply.
- Files: `docs/architecture.md`, `backend/src/features/` (not yet implemented)
- Cause: Fire-and-forget pattern with no retry queue, dead-letter mechanism, or timeout monitoring.
- Improvement path: For MVP, add structured logging with task IDs so failures are traceable. Post-MVP, consider a proper task queue (Celery, ARQ, or Redis Queue) with retry policies and failure notification to the student.

### Conversation memory loads last 20 messages from DB on every request

- Problem: `docs/chatbot.md` specifies `ConversationBufferWindowMemory (k=20)` with the note "Ao retomar sessao, carrega ultimas 20 mensagens do banco". This means every incoming WhatsApp message triggers a query to retrieve 20 chat_messages rows before the agent runs.
- Files: `docs/chatbot.md`, `docs/database.md` (`chat_messages` table)
- Cause: No caching layer for active session memory.
- Improvement path: Cache active session context in Redis for sessions active within the last N minutes. Fall back to DB query only for cold sessions.

### PGVector HNSW index parameters may need tuning for production load

- Problem: The HNSW index in `docs/database.md` is defined with default parameters `m=16, ef_construction=64`. These are reasonable defaults but may produce poor recall or slow build times as the knowledge base grows.
- Files: `docs/database.md` (HNSW index definition)
- Cause: Parameters are placeholders, not benchmarked against actual document volume.
- Improvement path: Benchmark search recall and latency after ingesting the full knowledge base. Adjust `ef_construction` upward (128+) if recall is insufficient.

## Fragile Areas

### `students.semester` is manually updated — no automated progression

- Files: `docs/database.md` (`students` table, `semester` column)
- Why fragile: The `semester` column is documented as "Atualizado manualmente via staff ou script de inicio de semestre". If this update is missed or run with errors, semester-dependent logic (available courses, prerequisite checks) will return incorrect results for all students.
- Safe modification: Any code touching `semester`-dependent queries must never assume the value is current. Add a validation endpoint that staff can run to verify semester data before academic period operations.
- Test coverage: No tests exist for this domain.

### Draft enrollment cancellation on period close is not atomic

- Files: `docs/chatbot.md`, `docs/database.md` (`enrollments.status`)
- Why fragile: When an enrollment period closes, all `draft` status enrollments should be set to `cancelled`. This is not described as a database trigger or scheduled job — it is implied to happen at period close time. If the process fails mid-run, some drafts remain as `draft` indefinitely while the period is `is_active = false`, creating an inconsistent state.
- Safe modification: Implement period close as a transaction that atomically sets `is_active = false` and bulk-updates all associated drafts to `cancelled` in a single transaction.
- Test coverage: No tests exist.

### `verification_codes` table has no cleanup automation in MVP

- Files: `docs/database.md` (Limpeza de Registros section)
- Why fragile: Expired and used codes accumulate indefinitely in the MVP. The cleanup is documented as a manual `docker exec` script. The `idx_verification_codes_email` index query is `(email, used, expires_at)` — as the table grows, auth performance degrades.
- Safe modification: The manual cleanup SQL is documented. Schedule it via cron on the host or via `pg_cron` (post-MVP roadmap item).
- Test coverage: None.

### Flutter `main.dart` uses invalid Dart syntax

- Files: `mobile/lib/main.dart` (lines 14, 49)
- Why fragile: `ColorScheme.fromSeed(...)` is written as `.fromSeed(...)` (missing class name) and `MainAxisAlignment.center` is written as `.center` (missing class name). The file uses abbreviated dot-access syntax that is not valid in Dart outside of enum context. This will cause compile errors before any project code is added.
- Safe modification: Fix both references before beginning any Flutter development.
- Test coverage: The only test (`mobile/test/widget_test.dart`) tests the counter widget — it will fail to compile for the same reason.
