    ```python
    DATABASE_URL: str = ""  # Set in __post_init__
    ```

    Then in `__post_init__` (since this is a frozen dataclass, use `object.__setattr__`):
    ```python
    def __post_init__(self) -> None:
        if not self.DATABASE_URL:
            from urllib.parse import quote_plus
            user = os.environ.get("POSTGRES_USER", "fcg3")
            password = os.environ.get("POSTGRES_PASSWORD", "change_me_in_production")
            host = os.environ.get("POSTGRES_HOST", "postgres")
            port = os.environ.get("POSTGRES_PORT", "5432")
            db = os.environ.get("POSTGRES_DB", "fcg3")
    ```

<task type="auto">
  <name>Task 2: Fix save_chat_message UUID generation for chat_messages.id</name>
  <files>ai_service/database.py</files>
  <action>
    **Root cause:** The `chat_messages` table has `id UUID NOT NULL` as primary key with NO `server_default` (the Alembic migration 006 sets `nullable=False` but no `DEFAULT gen_random_uuid()`). The SQLAlchemy ORM model uses `default=uuid.uuid4` which generates the UUID Python-side, but `ai_service/database.py` uses raw SQL and does NOT include `id` in the INSERT statement. Once the credential fix from Task 1 allows DB connections, every `save_chat_message` call will fail with a NOT NULL violation on the `id` column.

    **Fix `ai_service/database.py` `save_chat_message()`:**
    Update the INSERT to generate a UUID using `gen_random_uuid()` (PostgreSQL built-in since PG 13, which pg16 has):
    ```python
    def save_chat_message(pool: Any, session_id: str, role: str, content: str) -> None:
        """Persist a chat message for a session."""

        query = (
            "INSERT INTO chat_messages (id, chat_session_id, role, content) "
            "VALUES (gen_random_uuid(), %s, %s, %s)"
        )

        with pool.connection() as connection:
            with connection.cursor() as cursor:
                cursor.execute(query, (session_id, role, content))
            connection.commit()
    ```

    This uses `gen_random_uuid()` server-side rather than Python `uuid.uuid4()` to keep it consistent with how UUIDs are generated in the rest of the stack (the MCP server also uses PostgreSQL-side UUID generation). No Python import changes needed.

    **Verify the full /chat path works end-to-end after both fixes:**
    After the langchain-service is healthy (Task 1), test that `/chat` can persist messages by running:
