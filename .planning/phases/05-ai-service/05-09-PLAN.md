  <files>ai_service/ingest.py</files>
  <action>
    **Gap:** `IngestSettings.from_env()` at line 38 reads `DATABASE_URL` from `os.environ.get("DATABASE_URL")`. Plan 08 removed `DATABASE_URL` from the langchain-service Docker environment (replaced with `POSTGRES_*` component vars). The main `ai_service/config.py` `Settings` object handles this via `__post_init__` fallback, but `ingest.py` has its own independent env read that will fail with `RuntimeError("Missing required environment variables: DATABASE_URL")` in Docker. This was also identified as CR-01 in `05-REVIEW.md`.

    **Fix:** Replace the `from_env()` classmethod in `IngestSettings` (lines 36-58) to import and use `ai_service.config.settings.DATABASE_URL` instead of reading from `os.environ`. The replacement:

    ```python
    @classmethod
    def from_env(cls) -> "IngestSettings":
        from ai_service.config import settings as app_settings

        database_url = app_settings.DATABASE_URL
        openai_api_key = os.environ.get("OPENAI_API_KEY")

        if not openai_api_key:
            raise RuntimeError(
                "Missing required environment variable: OPENAI_API_KEY"
            )

        return cls(
            database_url=normalize_psycopg_dsn(database_url),
            openai_api_key=openai_api_key,
        )
    ```

    This reuses the `Settings` object which already handles `POSTGRES_*` component fallback, keeping a single source of truth for DATABASE_URL construction. The `OPENAI_API_KEY` check remains direct from `os.environ` because that key is always passed explicitly to the container and is not derived from components.

    **Do NOT change:** Any other part of `ingest.py` — the chunking logic, embedding pipeline, and storage flow are all correct.
