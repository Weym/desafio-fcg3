
pgvector cosine similarity query pattern:
```sql
SELECT content, source, category, 1 - (embedding <=> query_vec::vector) AS similarity
FROM knowledge_base_chunks
WHERE 1 - (embedding <=> query_vec::vector) >= 0.75
ORDER BY similarity DESC
LIMIT 3;
```
Note: pgvector `<=>` operator returns cosine DISTANCE (0 = identical, 2 = opposite).
Similarity = 1 - distance. Threshold >= 0.75 means distance <= 0.25.

**Design:** A factory function `create_rag_tool(db_pool, openai_api_key)` that returns a LangChain `@tool`-decorated function. The factory closure captures the pool and embeddings client.

**Implementation:**

1. **Embedding client (reused across queries):**
   ```python
   from langchain_openai import OpenAIEmbeddings
   embeddings = OpenAIEmbeddings(model="text-embedding-3-small", api_key=openai_api_key)
   ```
   Per D-17: fresh embedding per query, no caching. The `OpenAIEmbeddings` instance is reused but each `.embed_query()` call hits the API.

2. **Factory function:**
   ```python
   from langchain_core.tools import tool

   def create_rag_tool(db_pool, openai_api_key: str):
       embeddings = OpenAIEmbeddings(model="text-embedding-3-small", api_key=openai_api_key)

       @tool
       def search_knowledge_base(search_query: str) -> str:
           """Pesquisa a base de conhecimento academica para responder duvidas sobre regras, regulamentos, prazos e politicas do curso de Ciencia da Computacao. Use quando o aluno perguntar sobre regras de matricula, trancamento, prazos, curriculo ou regulamento academico."""
           # 1. Generate embedding for the search query
           query_vector = embeddings.embed_query(search_query)

           # 2. Format as pgvector string
           vector_str = str(query_vector)

           # 3. Query knowledge_base_chunks with cosine similarity
           with db_pool.connection() as conn:
               with conn.cursor() as cur:
                   cur.execute("""
                       SELECT content, source, category,
                              1 - (embedding <=> %s::vector) AS similarity
                       FROM knowledge_base_chunks
                       WHERE 1 - (embedding <=> %s::vector) >= 0.75
                       ORDER BY similarity DESC
                       LIMIT 3
                   """, [vector_str, vector_str])
                   rows = cur.fetchall()

           # 4. If no results above threshold, return empty (D-14)
           if not rows:
               return ""

           # 5. Format results as readable text for the agent
           chunks = []
           for content, source, category, similarity in rows:
               chunks.append(f"[Fonte: {source} | Categoria: {category} | Relevancia: {similarity:.2f}]\n{content}")

           return "\n\n---\n\n".join(chunks)

       return search_knowledge_base
   ```

3. **Key decisions:**
   - Per D-11: RAG is an explicit tool — the agent calls it when it decides to search docs.
   - Per D-12: The tool accepts `search_query` — the agent formulates the search (not raw user text).
   - Per D-13: Top 3 chunks above 0.75 threshold.
   - Per D-14: Empty string returned when no relevant chunks found. The agent's system prompt instructs how to handle this.
   - Per D-16: Same `text-embedding-3-small` model as ingest.
   - Per D-17: Fresh embedding per query (no cache).

4. **Tool description is critical:** The docstring must clearly describe WHEN to use this tool vs MCP tools. The agent uses this description to decide: policy/regulation questions → RAG; data queries/actions → MCP tools.

5. **psycopg3 note:** Use `db_pool.connection()` context manager for sync access. The `<=>` operator is pgvector's cosine distance operator. The vector parameter must be passed as a string representation `"[0.1, 0.2, ...]"` and cast to `::vector` in SQL.
