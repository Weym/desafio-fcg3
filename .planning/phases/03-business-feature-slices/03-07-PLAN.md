## Key Decisions
- **D-09:** Scheduling slots belong to resources (`scheduling_slots.resource_id` FK → `resources`). The `resources` table can represent staff, rooms, or other bookable entities.
- **D-10:** SELECT FOR UPDATE for appointment booking. Pessimistic lock on the slot during the booking transaction. Two students cannot reserve the same slot simultaneously.

## API Endpoints (from docs/api.md)
- `GET /scheduling/slots` — authenticated, MCP-accessible. Filters: date_from, date_to, staff_id. Defaults: today → today+7.
  **SM-03 (from 03-RESEARCH.md):** API uses `?staff_id=uuid` but DB has `resource_id`. Staff members must be represented as `resources` entries with `resource_type='staff'`. The `staff_id` query param maps to filtering resources where the resource represents that staff member. Slot response maps `resource` to `staff` in the API response shape.
- `POST /scheduling/slots` — staff only. Creates multiple slots from time range + duration.
- `POST /appointments` — student, MCP-accessible. Books a slot.
- `GET /appointments` — authenticated. Filters: student_id, status.
- `PUT /appointments/{id}/cancel` — student/staff, MCP-accessible. Cancels appointment.

## Database Tables
- `resources`: id, name, resource_type, capacity, location, is_available
- `scheduling_slots`: id, resource_id (FK → resources), date, start_time, end_time, is_available
- `appointments`: id, student_id, slot_id (FK → scheduling_slots), reason, status (scheduled/completed/cancelled/no_show)

    Create `SlotService`:

    1. `get_available_slots(db, date_from?, date_to?, resource_id?)` — APPT-01:
       - Default: date_from=today, date_to=today+7 days (per docs/api.md)
       - Query scheduling_slots where `is_available = True` AND `date BETWEEN date_from AND date_to`
       - If resource_id filter provided, filter by it
       - Join resources table for staff/resource name
       - Return list of SlotResponse

     2. `create_slots(db, resource_id: UUID, data: SlotCreate)` — APPT-STAFF-01:
       - Verify resource exists
       - **Overlap check (P-08 from 03-RESEARCH.md):** Before generating slots, query existing slots for the same resource_id + date. If any existing slot's time range overlaps with the requested range, reject with 409 `HORARIO_CONFLITANTE` or skip overlapping ranges.
       - Generate individual slots from the time range:
         - Start at `start_time`, create `slot_duration_minutes` slots until `end_time`
         - Example: start=08:00, end=12:00, duration=30min → 8 slots (08:00-08:30, 08:30-09:00, ..., 11:30-12:00)
       - Create each as a scheduling_slot with is_available=True
       - Return list of created slots

    Create `AppointmentService`:

    3. `book_appointment(db, student_id, data: AppointmentCreate)` — APPT-02, D-10:
       - **SELECT FOR UPDATE** on the slot:
         ```python
         slot = await db.execute(
             select(SchedulingSlot)
             .where(SchedulingSlot.id == data.slot_id)
             .with_for_update()
         )
         slot = slot.scalar_one_or_none()
         ```
       - If slot is None → 404 `SLOT_NAO_ENCONTRADO`
       - If slot.is_available is False → 409 `SLOT_JA_RESERVADO`
       - Set slot.is_available = False
       - Create appointment with status='scheduled'
       - Return AppointmentResponse

    4. `cancel_appointment(db, appointment_id, user)` — APPT-03:
       - Get appointment or 404
       - check_ownership (student can only cancel own)
       - Verify status == 'scheduled' (can't cancel completed/no_show)
       - Set appointment.status = 'cancelled'
       - Set slot.is_available = True (release the slot back)
       - Return updated appointment

    5. `list_appointments(db, params, student_id?, status?)` — APPT-04:
       - If role is student, force filter by own student_id
       - Apply status filter
       - Return paginated list

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Student → book appointment | Race condition: two students booking same slot |
| Student → cancel appointment | Student should only cancel own appointments |
| MCP → book/cancel | Service token needs ownership verification |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-27 | T (Tampering) | book appointment race | mitigate | SELECT FOR UPDATE (D-10) provides pessimistic locking; second concurrent request fails with 409 |
| T-03-28 | E (Elevation of Privilege) | cancel appointment | mitigate | check_ownership verifies student can only cancel own appointments |
| T-03-29 | T (Tampering) | slot creation | mitigate | require_staff prevents students from creating fake slots |
| T-03-30 | D (Denial of Service) | bulk slot creation | accept | Staff-only endpoint; low risk. MVP volume appropriate. |
</threat_model>
