## Key Decisions
- **D-01:** Build complete shared infrastructure BEFORE feature slices.
- **D-02:** Dual-auth: single `get_current_user_or_service()` — tries JWT Bearer first, then X-Service-Token. Returns user context in both cases.
- **D-03:** Error codes and messages all in Portuguese SCREAMING_SNAKE_CASE.
- **D-04:** Agent's discretion on IDOR check pattern.
- **D-05:** Same ownership check for JWT AND X-Service-Token (defense in depth).
- **D-06:** Staff role bypasses ownership check; student role requires resource.student_id == current_user.id.

## Dependencies from Phase 1 & 2
- Phase 1 created: SQLAlchemy models, async DB session factory in `backend/src/infrastructure/`.
- Phase 2 created: `get_current_user` dependency (JWT validation), `require_role` dependency, X-Service-Token middleware. These live in `backend/src/shared/` or `backend/src/features/auth/`.
- Phase 2 JWT payload: `sub` (user_id UUID), `role` (student | staff), `jti`, `name`, `email`.

## Standard API Conventions (from docs/api.md)
- Pagination: `?page=1&per_page=20&sort_by=created_at&order=desc`
- Response: `{"data": [...], "pagination": {"page": 1, "per_page": 20, "total": 150}}`
- Error: `{"error": {"code": "SCREAMING_SNAKE_CASE", "message": "...", "details": [...]}}`
- HTTP codes: 200, 201, 400, 401, 403, 404, 409, 422, 429, 500

    Create `get_current_user_or_service` FastAPI dependency per D-02:
    1. Check for `Authorization: Bearer {token}` header — if present, delegate to Phase 2's `get_current_user` to validate JWT and extract user context.

    3. If neither auth method succeeds, raise 401.

    The returned user context should be a Pydantic model or dataclass:
    ```python
    class UserContext:
        id: UUID           # user_id (from JWT sub) or student_id (from MCP)
        role: str          # "student", "staff", or "service"
        name: str | None
        email: str | None
    ```

    Create `check_ownership(resource_student_id: UUID, user: UserContext)` per D-05 and D-06:

    - Defense in depth per D-05: even service token requests go through ownership check.

    Create `require_staff(user: UserContext)` — raises ForbiddenException if role != "staff". For staff-only endpoints like student CRUD.

    Methods:
    - `async def list(self, db: AsyncSession, params: PaginationParams, filters: dict | None = None) -> tuple[list[T], int]` — Returns (items, total_count). Applies pagination (offset/limit from params), sorting (sort_by/order from params), and optional filters. Uses `select(Model).where(...).offset(...).limit(...)` pattern.
    - `async def get_by_id(self, db: AsyncSession, id: UUID) -> T | None` — Returns single entity or None.
    - `async def get_or_404(self, db: AsyncSession, id: UUID, resource_name: str) -> T` — Like get_by_id but raises `NotFoundException` with Portuguese message if not found.
    - `async def create(self, db: AsyncSession, data: dict) -> T` — Creates and returns entity. Calls `db.flush()` to get generated fields, then `db.refresh(instance)`.
    - `async def update(self, db: AsyncSession, instance: T, data: dict) -> T` — Updates fields from data dict. Only sets non-None values. Calls `db.flush()` + `db.refresh()`.

    The service should accept the SQLAlchemy model class in `__init__`:
    ```python
    class BaseService(Generic[T]):
        def __init__(self, model: type[T]):
            self.model = model
    ```

    All methods receive `db: AsyncSession` as first param (dependency injection pattern, not stored on self). This follows the async SQLAlchemy session-per-request pattern.

    Sorting should validate that `sort_by` is an actual column on the model to prevent SQL injection. If invalid column, default to `created_at`.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | S (Spoofing) | get_current_user_or_service | mitigate | JWT validated via Phase 2 middleware; X-Service-Token compared with hmac.compare_digest (constant-time) |
| T-03-02 | E (Elevation of Privilege) | check_ownership | mitigate | Every mutating endpoint calls check_ownership — even service token requests (D-05 defense in depth) |
| T-03-03 | T (Tampering) | sort_by parameter | mitigate | BaseService validates sort_by against actual model columns; rejects unknown columns |
| T-03-04 | I (Information Disclosure) | Error messages | mitigate | NotFoundException returns generic "não encontrado" — does not leak existence info for resources the user doesn't own |
| T-03-05 | E (Elevation of Privilege) | Role bypass | mitigate | require_staff dependency explicitly checks role; no implicit trust from auth status alone |
</threat_model>
