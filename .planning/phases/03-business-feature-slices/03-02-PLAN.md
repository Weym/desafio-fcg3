## API Endpoints (from docs/api.md)
- `GET /students` — staff only, paginated, filters: search, semester, status
- `GET /students/{id}` — authenticated (student+staff), IDOR-safe
- `POST /students` — staff only
- `PUT /students/{id}` — staff only
- `DELETE /students/{id}` — staff only (soft delete: status=inactive)
- `GET /students/{id}/academic-summary` — authenticated, accepts X-Service-Token (MCP)
- `GET /students/{id}/available-courses` — authenticated, accepts X-Service-Token (MCP)

## Database Tables (from docs/database.md)
- `students`: id, name, email, phone, registration_number, semester, status, enrollment_year, curriculum_id
- `grades`: student_id, course_id, grade_final, status (for CRA in academic-summary)
- `courses`: id, code, name, credits (for prerequisite filtering)
- `prerequisites`: course_id, prerequisite_id (for available-courses)
- `documents`: student_id, status (for pending_documents count)
- `appointments`: student_id, status (for next_appointment)

    4. `update_student(db, student_id, data: StudentUpdate)` — get_or_404, then BaseService.update.

    5. `soft_delete_student(db, student_id)` — get_or_404, set `status = "inactive"`, flush.

    6. `get_academic_summary(db, student_id)` — Queries:
       - Student basic info (name, semester, status)
       - Count completed courses: grades with `status = 'approved'` for this student
       - Total courses: count of curriculum_courses for student's curriculum
       - CRA/GPA: **Placeholder stub returning 0.0** — full CRA calculation is in Plan 03-05 (Grades slice, D-07/D-08). This method will be refined when grades service is implemented.
       - Pending documents: count of documents with `status NOT IN ('ready', 'delivered')` for this student
       - Next appointment: earliest appointment with `status = 'scheduled'` and `date >= today`
       Returns AcademicSummaryResponse.

    7. `get_available_courses(db, student_id)` — STU-07 with prerequisite filtering:
       - Get the student's curriculum_id
       - Get all courses in that curriculum via curriculum_courses
       - Get courses the student has already passed (grades.status = 'approved')
       - For each curriculum course, check if ALL prerequisites are in the passed set
       - Return only courses where `prerequisites_met = True` AND the student hasn't already passed them
       - Use a single efficient query or 2-3 queries (avoid N+1). Agent discretion on CTE vs Python-side filtering.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | E (Elevation of Privilege) | GET /students/{id} | mitigate | check_ownership verifies student can only view own data; staff bypasses (D-06) |
| T-03-07 | E (Elevation of Privilege) | POST/PUT/DELETE /students | mitigate | require_staff dependency blocks non-staff access |
| T-03-08 | T (Tampering) | POST /students (duplicate email) | mitigate | Unique constraint check before insert; ConflictException on duplicate |
| T-03-09 | I (Information Disclosure) | GET /students list | mitigate | Staff-only — students cannot enumerate other students |
| T-03-10 | E (Elevation of Privilege) | academic-summary via MCP | mitigate | Even with X-Service-Token, check_ownership still runs (D-05 defense in depth) |
