## Key Decisions
- **D-11:** Lock at TWO levels: (a) POST /enrollments/{id}/lock locks entire enrollment + all courses. (b) Individual enrollment_courses can also be locked independently. Both irreversible in MVP.
- **D-12:** Course drop (DELETE /enrollments/{id}/courses/{cid}) only in draft status. After confirmation, use lock instead.
- **D-13:** Outside active period → 409 PERIODO_MATRICULA_FECHADO. Prerequisite failure → 409 PREREQUISITO_NAO_CUMPRIDO.

## API Endpoints (from docs/api.md)
- `GET /enrollment-periods/current` — MCP-accessible
- `POST /enrollments` — student, MCP-accessible
- `POST /enrollments/{id}/confirm` — student, MCP-accessible
- `PUT /enrollments/{id}` — student
- `DELETE /enrollments/{id}/courses/{course_id}` — student, MCP-accessible
- `POST /enrollments/{id}/lock` — student, MCP-accessible
- `GET /enrollments` — authenticated
- `GET /staff/enrollment-periods` — staff only
- `POST /staff/enrollment-periods` — staff only
- `PUT /staff/enrollment-periods/{id}` — staff only

## Database Tables
- `enrollment_periods`: id, name, type, start_date, end_date, semester_year, is_active
- `enrollments`: id, student_id, enrollment_period_id, status (draft/confirmed/cancelled), confirmed_at
- `enrollment_courses`: id, enrollment_id, course_id, status (enrolled/dropped/locked)
- `prerequisites`: course_id, prerequisite_id
- `grades`: student_id, course_id, status (to check passed prerequisites)

## Enrollment Status Lifecycle
- `draft` → `confirmed` (via POST /confirm, during active period)
- `draft` → `cancelled` (if period closes)
- `confirmed` → `locked` (via POST /lock, irreversible)
- enrollment_courses: `enrolled` → `dropped` (only in draft), `enrolled` → `locked`

       - Verify `data.enrollment_period_id` matches active period
       - Verify no existing draft/confirmed enrollment for this student+period (409 `MATRICULA_JA_EXISTENTE`)
       - **Deduplicate course_ids** in the service layer before processing (P-06 from 03-RESEARCH.md): `course_ids = list(set(data.course_ids))` or raise 409 `DISCIPLINA_DUPLICADA` if duplicates found. DB UNIQUE constraint `(enrollment_id, course_id)` provides secondary defense.
       - For each course_id in data.course_ids:
         - Verify course exists (404 `DISCIPLINA_NAO_ENCONTRADA`)
         - **Prerequisite check (ENROLL-08):** Get all prerequisites for the course. Check that student has passed (grades.status = 'approved') ALL of them. If any missing → 409 `PREREQUISITO_NAO_CUMPRIDO` with details listing the missing prerequisite course name.

       - Return EnrollmentResponse

    6. `confirm_enrollment(db, enrollment_id, student_id)` — ENROLL-03:
       - **SELECT FOR UPDATE** on the enrollment row to prevent race condition (P-03 from 03-RESEARCH.md): two concurrent confirms could create duplicate grade records
       - Get enrollment or 404
       - check_ownership (IDOR)
       - Verify status == 'draft' (else 409 `MATRICULA_JA_CONFIRMADA`)
       - Verify enrollment period is still active (else 409 `PERIODO_MATRICULA_FECHADO`)
       - Set status = 'confirmed', confirmed_at = now()
       - Create grade records for each enrollment_course (status='in_progress', no grades yet)

    7. `update_enrollment_courses(db, enrollment_id, student_id, course_ids)` — ENROLL-04:
       - Verify status == 'draft'
       - Same prerequisite validation as create
       - Replace enrollment_courses

    8. `drop_course(db, enrollment_id, course_id, student_id)` — ENROLL-05:
       - Get enrollment, check_ownership
       - **D-12:** Verify enrollment status == 'draft'. If not draft → 409 `OPERACAO_NAO_PERMITIDA` with message "Remocao de disciplina so e permitida em matriculas em rascunho. Utilize trancamento (lock) para matriculas confirmadas."
       - Find enrollment_course, set status = 'dropped'

    9. `lock_enrollment(db, enrollment_id, student_id)` — ENROLL-06, D-11 level (a):
       - Get enrollment, check_ownership
       - Set enrollment.status = 'locked'
       - Set ALL enrollment_courses.status = 'locked' (the entire semester)
       - Irreversible — no status check needed (can lock from draft or confirmed)

    10. `list_enrollments(db, params, student_id?, semester_year?, status?)` — ENROLL-07:
        - If role is student, force filter by own student_id (IDOR-safe)
        - If role is staff, allow filtering by any student_id

    Note: Individual course locking (D-11 level b — trancamento parcial) can be added as an additional endpoint `POST /enrollments/{id}/courses/{cid}/lock` if needed, or handled within lock_enrollment logic. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Student → enrollment actions | Student could try enrolling in courses with unmet prereqs |
| Student → other student's enrollment | IDOR on enrollment_id |
| MCP → enrollment create/confirm | Service token must still validate period and prereqs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-14 | E (Elevation of Privilege) | POST /enrollments | mitigate | Prerequisite validation on every enrollment creation; cannot bypass via direct API call |
| T-03-15 | T (Tampering) | confirm enrollment | mitigate | Period active check + status check prevents double-confirmation and out-of-period confirmation |
| T-03-16 | E (Elevation of Privilege) | enrollment IDOR | mitigate | check_ownership on every mutating endpoint; student can only modify own enrollments |
| T-03-17 | R (Repudiation) | lock enrollment | mitigate | Lock is irreversible; once locked, cannot be unlocked (audit trail via status history) |
| T-03-18 | T (Tampering) | drop after confirm | mitigate | D-12: Drop only allowed in draft status; confirmed enrollments require lock (trancamento) |
| T-03-19 | D (Denial of Service) | duplicate enrollment | mitigate | Unique check prevents student from creating multiple enrollments for same period |
</threat_model>
