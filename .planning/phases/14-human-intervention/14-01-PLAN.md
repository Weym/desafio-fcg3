<threat_model>

## Trust Boundaries

| Boundary | Description |
| --- | --- |
| Staff → /chat-sessions/{id}/assign | Staff assumes conversation — must validate session is in correct state |
| Staff → /chat-sessions/{id}/reply | Staff sends message to student via WhatsApp — must validate ownership |
| Student → webhook | Student messages during human handling must NOT reach AI |
| AI response → escalation | Automatic escalation from AI response must be tamper-resistant |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
| --- | --- | --- | --- | --- |
| T-14-01 | Elevation of Privilege | POST /assign, /reply, /resolve | mitigate | `require_role('staff')` dependency on all 3 endpoints — students get 403 |
| T-14-02 | Spoofing | POST /reply — wrong staff | mitigate | Validate `session.assigned_staff_id == current_user.id` before allowing reply (per D-09) |
| T-14-03 | Tampering | Status transition bypass | mitigate | Validate session.status before state transitions: assign requires 'human_needed', reply/resolve require 'human_active'. Return 409 on invalid. |
| T-14-04 | Information Disclosure | GET /interventions | mitigate | Staff sees all 'human_needed' sessions + only their own 'human_active' — no cross-staff data leakage |
| T-14-05 | Denial of Service | Escalation keyword spam | accept | Low risk — escalation only changes session status once (already in human_needed = no-op on repeat). Natural rate limiting via WhatsApp API. |
| T-14-06 | Repudiation | Staff reply not attributable | mitigate | assigned_staff_id records who handled the session; messages saved with created_at timestamp |

</threat_model>
