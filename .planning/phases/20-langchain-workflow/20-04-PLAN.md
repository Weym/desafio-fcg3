Per D-13 and D-14: The current flow blocks ALL unverified students from reaching the agent. Modify to implement lazy OTP:

**Current flow:**
```
message → phone lookup → session → if unverified: verification_flow → if verified: process_verified_message
```

**New flow (lazy OTP):**
```
message → phone lookup → session → if unverified OR verified: process_message (agent handles both)
                                  → if awaiting_email or awaiting_code: verification_flow (OTP in progress)
```

**In `backend/src/features/webhook/router.py`:**

Find the routing logic where `verification_state` determines flow. Change it so:
- `verification_state == "unverified"`: Now routes to agent (was: starts verification flow)
- `verification_state == "awaiting_email"` or `"awaiting_code"`: Still routes to verification flow (OTP in progress)
- `verification_state == "verified"`: Routes to agent (unchanged)

The key change: `unverified` sessions now go to the agent instead of the verification state machine. The verification flow is only active when OTP is already in progress (`awaiting_email`/`awaiting_code`).

**In `backend/src/features/webhook/background.py`:**

Rename `process_verified_message` to `process_message` (or add alias) since it now handles both verified and unverified students. The function signature does not change — it still receives session_id, message_text, phone, wa_client.

Add a note/docstring clarifying: "Handles both verified and unverified students. The AI agent + MCP layer gates mutating actions — if the student tries a mutating action while unverified, the agent will request verification mid-conversation per D-15."

Per D-16: Once OTP is completed mid-conversation, `verification_state='verified'` persists for the entire session. This is already the case — `session.verification_state` is stored in the DB.

Per D-14: Read-only MCP tools trust phone-based identity. The phone→student lookup already provides student_id which is injected by the MCP server via session context — no changes needed for read operations.

Per D-15: When the agent needs to execute a mutating action and the student isn't verified, the agent will naturally request verification ("Para executar essa acao, preciso confirmar sua identidade. Qual seu email institucional?"). This response comes from the AI agent based on the system prompt instruction (D-21).

When the student responds with their email (detected by the existing EMAIL_REGEX), the webhook needs to handle the transition from normal conversation to verification flow.

Add a method `initiate_mid_conversation_verification` to `WebhookService`:

```python
async def initiate_mid_conversation_verification(
    self, session: ChatSession, db: AsyncSession
) -> None:
    """Transition session to awaiting_email state for mid-conversation OTP.

    Called when the agent requests verification for a mutating action.
    The student's next message will be processed by the verification flow.
    Per D-15/D-16.
    """
    session.verification_state = "awaiting_email"
    await db.flush()
```

Also, modify the `handle_verification_flow` method's "unverified" state handling:
- Remove the automatic "ask for email" behavior from the `unverified` state (since unverified students now go to the agent directly)
- The `unverified` state should no longer be reached by `handle_verification_flow` — only `awaiting_email` and `awaiting_code` states reach this function

Update the routing logic in the service: if `verification_state == "unverified"`, the message goes to the agent. Only `awaiting_email` and `awaiting_code` trigger `handle_verification_flow`.

Per D-16: Once verified, the state persists for the session. The existing code already stores `verification_state = 'verified'` in the database on successful OTP — no change needed.

NOTE: The MCP server's middleware (in `mcp_server/`) already resolves student_id from the chat session. For mutating tools called while unverified, the MCP server should return an error that the agent interprets as "verification needed." This is handled by the agent's system prompt instruction (D-21) which tells it to check verification status before calling mutating tools. The code-level safety net is in the MCP middleware — if somehow a mutating tool is called while unverified, the MCP server returns a structured error.
