Fix unverified student reads being blocked (UAT Test 5) and broken OTP flow (UAT Test 6) by implementing the MCP middleware verification gate and rewriting the system prompt for lazy OTP.

Add the `_enforce_verification_gate` method to the class:

```python
async def _enforce_verification_gate(
    self, context: MiddlewareContext, session_data: dict
) -> None:
    """Block mutating tool calls for unverified students (D-15/D-21).

    Read-only tools (readOnlyHint=True) are allowed for all students.
    Mutating tools (no readOnlyHint) require verification_state='verified'.
    """
    verification_state = session_data.get("verification_state", "unverified")
    if verification_state == "verified":
        return  # Verified students can use all tools

    # Check if the tool is read-only via annotations
    tool_name = context.message.name
    try:
        tool = await context.fastmcp_context.fastmcp.get_tool(tool_name)
        annotations = getattr(tool, "annotations", None)
        if annotations and getattr(annotations, "readOnlyHint", False):
            return  # Read-only tool, allowed for unverified students
    except Exception:
        pass  # If we can't determine, block by default (safe side)

    # Mutating tool + unverified student → block with actionable error
    from fastmcp.exceptions import ToolError
    raise ToolError(
        f"Acao bloqueada: o aluno precisa verificar sua identidade antes de executar '{tool_name}'. "
        "Solicite que o aluno informe seu email institucional para receber o codigo de verificacao."
    )
```

Import `ToolError` at the top of middleware.py (add to existing imports from fastmcp or import inline as shown).

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LangChain agent → MCP Server | Agent calls tools; MCP middleware gates mutating tools based on verification_state from DB |
| Backend → AI Service | verification_state passed as HTTP field; AI service trusts backend's value |
| Agent prompt → Tool selection | System prompt instructs agent behavior; MCP middleware is the enforcement layer |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-11-01 | Elevation of Privilege | MCP middleware | mitigate | Middleware reads verification_state directly from chat_sessions DB table (authoritative source), not from AI service request. Even if agent/prompt is manipulated, DB-level check prevents mutating tool execution. |
| T-20-11-02 | Spoofing | verification_state in HTTP body | accept | Backend is the sole caller of AI service (X-Service-Token validated). The verification_state in HTTP body is informational for the agent's prompt context only — the actual gate is in MCP middleware reading from DB. Defense in depth. |
| T-20-11-03 | Tampering | system prompt | mitigate | Rule #9 is defense-in-depth only. Even if LLM ignores the prompt and calls a mutating tool, MCP middleware blocks it. Prompt injection defense (4 layers from Plan 05) remains active. |
| T-20-11-04 | Information Disclosure | ToolError message | accept | Error message tells agent to ask for email — this is the intended UX. No internal details leaked (no API URLs, table names, etc.). |

</threat_model>
