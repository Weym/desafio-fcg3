**mcp_server/dependencies.py** — per D-04/D-05/D-06/D-07:
Create two dependency functions:

1. `resolve_chat_session_id(request: Request = CurrentRequest()) -> str`:
   - Reads `X-Chat-Session-ID` header from Starlette Request.
   - If missing/empty → raise `ToolError("Sessao invalida, nao foi possivel identificar o aluno.")`.
   - Returns the UUID string.

2. `resolve_student_id(request: Request = CurrentRequest()) -> str`:
   - Calls `resolve_chat_session_id` internally.
   - Gets `db_pool` from lifespan context via `get_context()` → `ctx.lifespan_context["db_pool"]`.
   - Runs: `SELECT student_id FROM chat_sessions WHERE id = $1 AND status = 'active'` with the session_id.
   - If no row found → raise `ToolError("Sessao invalida, nao foi possivel identificar o aluno.")`.
   - Returns `str(student_id)`.

Both use `from fastmcp.dependencies import CurrentRequest` and `from fastmcp.server.dependencies import get_context`.

**mcp_server/api_client.py** — per D-08/D-09/D-18 and retry spec from docs/mcp.md:
Create `async def call_api(client: httpx.AsyncClient, method: str, path: str, **kwargs) -> dict`:
1. Try `response = await client.request(method, path, **kwargs)` + `response.raise_for_status()`.
2. On `httpx.HTTPStatusError` where `status_code < 500` → parse error JSON → translate to Portuguese text → raise `ToolError("Erro: {translated_message}")`. If error JSON has `error.message`, use it directly (already in Portuguese per Phase 3 D-03).
3. On `httpx.HTTPStatusError` where `status_code >= 500` or `httpx.TimeoutException` → retry once immediately. If retry succeeds → return response, mark as retried. If retry fails → raise `ToolError("Erro interno do servidor. Tente novamente mais tarde.")`.
4. Return `(response_data: dict, retried: bool)` tuple — the middleware needs `retried` for the log record.

Also create `async def call_api_raw(...)` that returns the raw httpx.Response for edge cases.

**mcp_server/middleware.py** — per D-10/D-11/D-12:
Create `class ToolLoggingMiddleware(Middleware)`:
1. `__init__(self)` — no args (gets db_pool from context at runtime).
2. `async def on_call_tool(self, context: MiddlewareContext, call_next)`:
   - Extract `tool_name = context.message.name`, `input_params = context.message.arguments or {}`.
   - Get `chat_session_id` from HTTP headers via `get_http_headers()` → `headers.get("x-chat-session-id")`.
   - Record `start = time.monotonic()`.
   - Try `result = await call_next(context)` → `status = "success"`.
   - On exception → `status = "error"`, `output_result = None`.
   - Calculate `latency_ms = int((time.monotonic() - start) * 1000)`.
   - Get `db_pool` from `context.fastmcp_context.lifespan_context["db_pool"]`.
   - `INSERT INTO mcp_action_logs (chat_session_id, tool_name, input_params, output_result, reasoning, latency_ms, retry, status)` VALUES. Use `json.dumps()` for JSONB fields. `reasoning` = NULL per D-11.
   - `retry` field: Check if the tool result indicates retry happened (tools will set a flag via context state or the api_client return). Default to `false`.
   - If status was "error" after logging, re-raise the exception.
   - Return result.

Note: The `retry` and `retry_success` status detection will be refined — the api_client sets retry info on a context-local variable. For now, default retry=false, status=success|error. The tool implementations in Plans 02/03 will use `call_api` which returns retry info.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AI Service → MCP Server | Untrusted: AI agent sends tool call requests via streamable-http. X-Chat-Session-ID header could be spoofed if AI service is compromised. |
| MCP Server → FastAPI | Internal service auth via X-Service-Token. Token must be constant-time compared. |
| MCP Server → PostgreSQL | Direct DB access for session resolution and logging. SQL injection risk on session_id. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | X-Chat-Session-ID header | mitigate | Validate UUID format before DB query; only accept active sessions from chat_sessions table |
| T-04-02 | Tampering | Tool input_params | mitigate | FastMCP validates params against schema; asyncpg parameterized queries prevent SQL injection |
| T-04-03 | Information Disclosure | student_id in logs | mitigate | student_id NEVER in input_params or tool schemas; only used internally for API path construction |
| T-04-04 | Information Disclosure | MCP_SERVICE_TOKEN | mitigate | Never in source code; environment variable only; hmac.compare_digest on FastAPI side |
| T-04-05 | Denial of Service | Unbounded tool calls | accept | MVP volume is low; rate limiting can be added via FastMCP middleware if needed |
| T-04-06 | Elevation of Privilege | IDOR via session_id | mitigate | student_id resolved server-side only; never accepted from agent input |
</threat_model>
